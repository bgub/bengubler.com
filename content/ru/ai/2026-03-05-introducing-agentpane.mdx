# Представляем agentpane

agentpane — это локальный веб-интерфейс для ИИ-агентов, помогающих писать код. Вы запускаете `npx agentpane`, интерфейс открывается в браузере, и вы можете общаться с Claude Code, Codex или любым агентом, который работает по [ACP (Agent Client Protocol)](https://github.com/anthropics/agent-protocol), — и всё это в многопанельном интерфейсе с потоковым выводом, сохранением состояния и управлением сессиями:

```bash
npx agentpane
# → http://localhost:6767
```

Выберите агента, укажите рабочую директорию и запустите сессию. Агент работает на вашем компьютере как подпроцесс. Никакого облака, никакого развёртывания, никаких аккаунтов.

## Несколько панелей, несколько агентов

Главное, что мне было нужно, — это возможность запускать несколько агентов одновременно и видеть их рядом. agentpane позволяет открыть до четырёх панелей с изменяемым размером и вкладками. Можно перетаскивать сессии с боковой панели в любую из них, а вкладку — перетащить, чтобы открыть новую разделённую панель. Раскладка сохраняется даже после обновления страницы.

Это значит, что в одной панели у вас может быть Claude Code, работающий над бэкендом, а в другой — Codex, рефакторящий фронтенд, — всё на одном экране, в одном приложении.

![Две сессии, запущенные рядом — Codex слева, Claude Code справа](/blog-images/agentpane-hero.png)

## Архитектура

Приложение состоит из двух процессов:

* **API-сервер** (Hono, порт 3456) — запускает агентов, управляет сессиями и отвечает за сохранение данных
* **Веб-фронтенд** (Next.js, порт 6767) — пользовательский интерфейс, в котором вызовы `/api` перенаправляются на бэкенд

Бэкенд построен на [Effect.ts](https://effect.website/) и использует внедрение зависимостей через компонуемые сервисные слои:

```
SessionRepo (DB)
  → ConnectionManager (agent subprocess lifecycle)
  → PromptEngine (turn orchestration)
  → EventHub / EventBroadcaster (SSE)
  → WriteQueue (batched DB writes)
```

Агенты взаимодействуют через ACP — JSON-RPC 2.0 поверх stdio. Когда вы отправляете запрос, `PromptEngine` создаёт записи ходов диалога, передаёт сообщение подпроцессу агента через ACP и передаёт результаты обратно через поток Server-Sent Events.


### Потоковая передача с безопасным переподключением

Фронтенд подписывается на SSE-эндпоинт для каждой сессии. Каждое событие получает монотонно возрастающий ID. `EventBroadcaster` хранит кольцевой буфер (512KB / 1000 событий), поэтому, если браузер отключится и снова подключится, он досылает пропущенные события с помощью заголовка `Last-Event-ID`. Никаких пропусков в диалоге и никаких ручных обновлений.

### Отказоустойчивое сохранение

Всё состояние записывается в локальную базу данных SQLite по пути `~/.agentpane/agentpane.db`. Но запись происходит не сразу — `WriteQueue` накапливает операции и пакетно сбрасывает их каждые 50 мс. Перед сбросом операции сохраняются в таблице восстановления. Если сервер аварийно завершится в середине сброса, после перезапуска он продолжит с того места, где остановился. Сообщения не теряются.

```sql
-- Таблица восстановления
CREATE TABLE write_queue_ops (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  op_json TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```


## Что видят агенты

Для agentpane неважно, какая модель лежит в основе агента — он общается по ACP. Когда вы подключаетесь к сессии, `ConnectionManager`:

1. Определяет исполняемый файл агента (просматривает `node_modules/.bin/` для совместимости с npm)
2. Запускает его как дочерний процесс через каналы stdio
3. Согласовывает возможности ACP (auth, config, modes, commands)
4. Пытается возобновить сессию при повторном подключении к ранее открытой сессии

Интерфейс подстраивается под возможности агента. Если агент предоставляет параметры конфигурации, они отображаются в виде выпадающих списков. Если он поддерживает слеш-команды, вы получаете автодополнение. Если он потоково передаёт блоки размышлений, они отображаются как сворачиваемые секции.

## Фронтенд

Веб-приложение построено на Next.js 16, React 19 и React Compiler — ручное использование `useMemo` или `useCallback` нигде не требуется. Состояние разделено на два слоя контекста:

* **SessionProvider** — активная сессия, проверки работоспособности, CRUD-операции с сессиями
* **LayoutProvider** — конфигурация панелей, порядок вкладок, drag-and-drop

Серверное состояние (сессии, диалоги, использование токенов) управляется через TanStack React Query и сбрасывается при SSE-событиях. Markdown рендерится по мере поступления данных с помощью [Streamdown](https://github.com/anthropics/streamdown) с подсветкой синтаксиса от Shiki.
