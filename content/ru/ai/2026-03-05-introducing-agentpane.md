# Представляем agentpane

agentpane — это локальный веб-интерфейс для ИИ-агентов для программирования. Вы запускаете `npx agentpane`, он открывается в браузере, и вы можете общаться с Claude Code, Codex или любым агентом, поддерживающим [ACP (Agent Client Protocol)](https://github.com/anthropics/agent-protocol), — всё это в многопанельном интерфейсе со стримингом, сохранением состояния и управлением сессиями:

```bash
npx agentpane
# → http://localhost:6767
```

Выберите агента, рабочую директорию и запустите сессию. Агент работает на вашем компьютере в виде подпроцесса. Никакого облака, никакого развёртывания, никаких аккаунтов.

![Экран настройки agentpane с выбором агента и директории](/blog-images/agentpane-setup.png)

## Многопанельный, мультиагентный интерфейс

Главное, чего я хотел, — это возможность запускать сразу нескольких агентов и видеть их рядом. agentpane позволяет открыть до четырёх панелей с вкладками и изменяемым размером. Можно перетаскивать сессии из боковой панели в любую из них или перетащить вкладку, чтобы открыть новую панель. Раскладка сохраняется даже после обновления страницы.

То есть в одной панели Claude Code может работать над вашим бэкендом, а в другой Codex — рефакторить фронтенд: один экран, одно приложение.

![Две сессии, запущенные бок о бок — Codex слева, Claude Code справа](/blog-images/agentpane-hero.png)

## Архитектура

Приложение состоит из двух процессов:

* **API-сервер** (Hono, порт 3456) — запускает агентов, управляет сессиями и отвечает за сохранение состояния
* **Веб-фронтенд** (Next.js, порт 6767) — UI, где вызовы `/api` перенаправляются на бэкенд

Бэкенд построен на [Effect.ts](https://effect.website/) и использует внедрение зависимостей через компонуемые слои сервисов:

```
SessionRepo (DB)
  → ConnectionManager (agent subprocess lifecycle)
  → PromptEngine (turn orchestration)
  → EventHub / EventBroadcaster (SSE)
  → WriteQueue (batched DB writes)
```

Агенты взаимодействуют по ACP — JSON-RPC 2.0 поверх stdio. Когда вы отправляете промпт, `PromptEngine` создаёт записи шагов диалога, передаёт сообщение подпроцессу агента через ACP и передаёт результаты обратно через Server-Sent Events в потоковом режиме.

### Стриминг с безопасным переподключением

Фронтенд подписывается на SSE-эндпоинт для каждой сессии. Каждое событие получает монотонно возрастающий ID. `EventBroadcaster` хранит кольцевой буфер (512KB / 1000 событий), поэтому, если ваш браузер отключится и затем переподключится, пропущенные события будут воспроизведены с помощью заголовка `Last-Event-ID`. Никаких пропусков в диалоге и никакого ручного обновления.

### Сохранение данных с защитой от сбоев

Всё состояние записывается в локальную базу данных SQLite по пути `~/.agentpane/agentpane.db`. Но запись происходит не сразу: `WriteQueue` накапливает операции и пакетно сбрасывает их каждые 50 мс. Перед этим операции сохраняются в таблице восстановления. Если сервер аварийно завершится во время сброса, после перезапуска он продолжит с того места, где остановился. Сообщения не теряются.

```sql
-- Таблица восстановления
CREATE TABLE write_queue_ops (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  op_json TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## Что видят агенты

Для agentpane не важно, какая модель лежит в основе агента — он работает по ACP. Когда вы подключаетесь к сессии, `ConnectionManager`:

1. Определяет исполняемый файл агента (просматривает `node_modules/.bin/` для совместимости с npm)
2. Запускает его как дочерний процесс с каналами stdio
3. Согласовывает возможности ACP (аутентификация, конфигурация, режимы, команды)
4. Пытается возобновить сессию при повторном подключении к ней

UI подстраивается под то, что поддерживает агент. Если агент предоставляет параметры конфигурации, они отображаются в виде выпадающих списков. Если он поддерживает `slash commands`, вы получаете автодополнение. Если он передаёт блоки рассуждений в потоковом режиме, они отображаются как сворачиваемые разделы.

## Фронтенд

Веб-приложение построено на Next.js 16 с React 19 и React Compiler — ручной `useMemo` и `useCallback` не используются вообще. Состояние разделено на два слоя контекста:

* **SessionProvider** — активная сессия, проверки состояния, CRUD для сессий
* **LayoutProvider** — конфигурация панелей, порядок вкладок, drag-and-drop

Серверное состояние (сессии, разговоры, использование токенов) управляется через TanStack React Query и инвалидируется по событиям SSE. Markdown рендерится по мере поступления через [Streamdown](https://github.com/anthropics/streamdown) с подсветкой синтаксиса от Shiki.
