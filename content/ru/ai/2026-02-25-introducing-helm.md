# Знакомьтесь: helm

helm — это типизированный фреймворк на TypeScript для ИИ-агентов. Вместо того чтобы обращаться к оболочке и разбирать строки, агенты вызывают типизированные функции со структурированными входными и выходными данными:

```ts
import { createHelm, git, fs, grep } from "@bgub/helm";

const agent = createHelm({
  permissions: {
    "fs.read": "allow",
    "fs.write": "ask",
    "fs.remove": "deny",
    "git.status": "allow",
    "git.*": "ask",
  },
  onPermissionRequest: async (operation, args) => {
    return confirm(`Allow ${operation}?`);
  },
})
  .use(fs())
  .use(git())
  .use(grep());

const { staged, unstaged, branch } = await agent.git.status();
const { content } = await agent.fs.read("./package.json");
```

Вы регистрируете &quot;навыки&quot; — группы связанных операций — с помощью паттерна Builder. TypeScript выводит полный тип на каждом шаге. У каждой операции есть уровень разрешения: `allow` (выполняется сразу), `ask` (приостанавливается и ждёт подтверждения) или `deny` (выбрасывает `PermissionDeniedError`). Разрешения определяются по приоритету: точное совпадение → подстановочный знак → значение по умолчанию, заданное автором навыка → глобальное значение по умолчанию.

В helm есть встроенные навыки для повседневных задач агентов: `fs`, `git`, `grep`, `edit`, `shell`, `http`. Вы можете определить собственные навыки, и они сразу получат типы, поиск и разрешения.

## Демонстрация

Я сделал чат-бота, у которого у агента есть ровно два инструмента: `search` и `execute`.

`search` выполняет поиск по ключевым словам среди всех зарегистрированных операций helm — агент вызывает его, чтобы понять, что доступно, и посмотреть сигнатуры функций:

```ts
agent.search("file read");
// → [{ qualifiedName: "fs.read",
//      description: "Read a file and return its content as a string",
//      signature: "(path: string) => Promise<{ content: string }>", ... }]
```

`execute` принимает произвольный JavaScript-код и выполняет его через API агента helm. LLM пишет код примерно такого вида:

```js
const { staged, unstaged, branch } = await agent.git.status();
return { branch, staged: staged.length, unstaged: unstaged.length };
```

Два инструмента в контексте — независимо от того, сколько навыков зарегистрировано. Агент сам находит всё необходимое по мере надобности и пишет код для работы с ним.

### Изоляция недоверенного кода

Инструмент `execute` запускает любой JavaScript, который генерирует LLM. Чтобы сделать это безопасно, в демо код выполняется в песочнице с помощью [SES (Secure ECMAScript)](https://github.com/endojs/endo/tree/master/packages/ses) в дочернем процессе.

`lockdown()` в SES замораживает все встроенные сущности JavaScript — `Object`, `Array`, `Promise`, `Function` и всё остальное. Код выполняется внутри `Compartment` — изолированной глобальной области видимости, у которой есть доступ ровно к двум вещам: прокси `agent` и заглушке `console`. `fetch`, `require`, `import`, `process`, `fs` — ничего из этого в `Compartment` нет. Единственный способ сделать что-то полезное — использовать прокси `agent`.

`agent` внутри песочницы — это не настоящий агент helm, а рекурсивный `Proxy`, который перехватывает доступ к свойствам и вызовы функций. Когда код вызывает `agent.git.status()`, прокси отправляет IPC-сообщение родительскому процессу. Родительский процесс вызывает реальный метод настоящего агента helm, выполняет полную проверку разрешений и отправляет результат обратно. Если для операции задано значение `"ask"`, родительский процесс приостанавливается и ждёт подтверждения пользователя, прежде чем ответить. Если задано `"deny"`, ошибка возвращается обратно через IPC.

Код в песочнице даже не подозревает, что всё это происходит. Он просто видит, что его `await` успешно возвращает значение. Единственный способ взаимодействовать с внешним миром — через операции helm, ограниченные системой разрешений.

### Интерфейс разрешений

В интерфейсе чата есть боковая панель со списком всех зарегистрированных навыков и операций; у каждой — переключатель allow/ask/deny. Изменение разрешения вступает в силу со следующего сообщения.

![Панель инструментов с настройками разрешений для каждой операции](/blog-images/helm-demo-app.png)

Когда LLM доходит до операции, для которой задано `"ask"`, сервер отправляет на фронтенд запрос на подтверждение. У вызова инструмента появляется встроенный баннер с кнопками Allow и Deny. Сервер ждёт `Promise<boolean>`, пока пользователь не нажмёт одну из них.

![Вызов инструмента execute, ожидающий подтверждения пользователя с кнопками Allow и Deny](/blog-images/helm-demo-app-sandbox-feature.png)

Если пользователь отказывает, `PermissionDeniedError` передаётся обратно до самого конца, и LLM видит это в результате выполнения инструмента. Она может объяснить, зачем ей нужно это разрешение, попробовать другой подход или отказаться от попытки.

## Вдохновение

Эта архитектура — когда агенту дают инструмент для выполнения кода вместо десятков отдельных инструментов — вдохновлена [code mode](https://blog.cloudflare.com/code-mode-mcp/) от Cloudflare для их MCP-сервера: там им удалось сократить расход токенов на 99,9%, заменив более 2 500 инструментов для API-эндпоинтов на `search` + `execute`. [Похожая идея Rhys Sullivan](https://x.com/RhysSullivan/status/2019819177473933404) помогла мне окончательно оформить эту мысль: сочетание выполнения кода, возможности находить нужное и тонкой модели разрешений означает, что агент может делать что угодно, но не может выйти за рамки.
