# Знакомьтесь: helm

helm — это типизированный фреймворк на TypeScript для ИИ-агентов. Вместо запуска shell-команд и разбора строк агенты вызывают типизированные функции со структурированными входными и выходными данными:

```ts
import { createHelm, git, fs, grep } from "@bgub/helm";

const agent = createHelm({
  permissions: {
    "fs.read": "allow",
    "fs.write": "ask",
    "fs.remove": "deny",
    "git.status": "allow",
    "git.*": "ask",
  },
  onPermissionRequest: async (operation, args) => {
    return confirm(`Allow ${operation}?`);
  },
})
  .use(fs())
  .use(git())
  .use(grep());

const { staged, unstaged, branch } = await agent.git.status();
const { content } = await agent.fs.read("./package.json");
```

Вы регистрируете &quot;навыки&quot; — группы связанных операций — с помощью паттерна Builder. TypeScript выводит полный тип на каждом шаге. У каждой операции есть уровень разрешения: `allow` (выполняется сразу), `ask` (приостанавливается до подтверждения) или `deny` (вызывает `PermissionDeniedError`). Разрешения применяются в порядке приоритета: точное совпадение → шаблон с подстановкой → значение по умолчанию, заданное автором навыка → глобальное значение по умолчанию.

helm поставляется со встроенными навыками для повседневных задач агентов: `fs`, `git`, `grep`, `edit`, `shell`, `http`. Вы можете определять собственные навыки, и они автоматически получают типы, search и разрешения.

## Демо

Я создал чат-бота, у которого есть ровно два инструмента: `search` и `execute`.

`search` выполняет поиск по ключевым словам среди всех зарегистрированных операций helm — агент вызывает его, чтобы узнать, какие возможности доступны, и изучить сигнатуры функций:

```ts
agent.search("file read");
// → [{ qualifiedName: "fs.read",
//      description: "Read a file and return its content as a string",
//      signature: "(path: string) => Promise<{ content: string }>", ... }]
```

`execute` принимает произвольный JavaScript-код и выполняет его через API агента helm. LLM пишет такой код:

```js
const { staged, unstaged, branch } = await agent.git.status();
return { branch, staged: staged.length, unstaged: unstaged.length };
```

В контексте — всего два инструмента, независимо от того, сколько навыков зарегистрировано. Агент по мере необходимости сам понимает, что ему нужно, и пишет код, чтобы это использовать.

### Изоляция недоверенного кода

Инструмент `execute` запускает любой JavaScript, который напишет LLM. Чтобы сделать это безопасным, в демо этот код выполняется в песочнице с помощью [SES (Secure ECMAScript)](https://github.com/endojs/endo/tree/master/packages/ses) в дочернем процессе.

`lockdown()` в SES замораживает все встроенные объекты JavaScript — `Object`, `Array`, `Promise`, `Function`, вообще всё. Код выполняется внутри `Compartment` — изолированной глобальной области видимости, у которой есть доступ ровно к двум вещам: прокси `agent` и заглушке `console`. `fetch`, `require`, `import`, `process`, `fs` — ничего этого в `Compartment` нет. Единственный способ сделать что-то интересное — использовать прокси агента.

`agent` внутри песочницы — не настоящий агент helm, а рекурсивный `Proxy`, который перехватывает обращения к свойствам и вызовы функций. Когда код вызывает `agent.git.status()`, прокси отправляет IPC-сообщение родительскому процессу. Тот вызывает настоящий метод у настоящего агента helm, выполняет полную проверку разрешений и отправляет результат обратно. Если для операции задано значение `"ask"`, родительский процесс приостанавливается и ждёт подтверждения пользователя перед ответом. Если задано `"deny"`, ошибка возвращается обратно через IPC.

Код в песочнице даже не подозревает, что всё это происходит. Он просто получает значение при `await`. Единственный способ взаимодействовать с внешним миром — через операции helm, защищённые системой разрешений.

### Интерфейс управления разрешениями

В интерфейсе чата есть боковая панель со списком всех зарегистрированных навыков и операций; у каждой из них есть переключатель allow/ask/deny. Изменение разрешения вступает в силу со следующего сообщения.

![Панель инструментов с элементами управления разрешениями для каждой операции](/blog-images/helm-demo-app.png)

Когда LLM доходит до операции, для которой установлено значение `"ask"`, сервер отправляет во frontend запрос на подтверждение. Вызов инструмента показывает встроенный баннер с кнопками Allow и Deny. Сервер ждёт `Promise<boolean>`, пока пользователь не нажмёт одну из них.

![Вызов инструмента execute в ожидании подтверждения пользователя с кнопками Allow и Deny](/blog-images/helm-demo-app-sandbox-feature.png)

Если пользователь отклоняет запрос, `PermissionDeniedError` передаётся обратно до самого конца, и LLM видит это в результате работы инструмента. Она может объяснить, зачем ей нужно это разрешение, попробовать другой подход или отказаться от попытки.

## Вдохновение

Эта архитектура — дать агенту инструмент для выполнения кода вместо десятков отдельных инструментов — вдохновлена [code mode](https://blog.cloudflare.com/code-mode-mcp/) от Cloudflare для их MCP-сервера: там они сократили расход токенов на 99,9%, заменив более 2 500 инструментов с API-эндпоинтами на `search` + `execute`. [Похожая идея Rhys Sullivan](https://x.com/RhysSullivan/status/2019819177473933404) помогла мне окончательно оформить эту мысль: сочетание выполнения кода, обнаружения возможностей и гибкой модели разрешений позволяет агенту делать что угодно, не теряя управляемости.
