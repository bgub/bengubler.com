Восемь лет назад я выпустил свою первую библиотеку TypeScript с открытым исходным кодом — [Squirrelly](https://github.com/squirrellyjs/squirrelly) — и в ней было всего два файла: `package.json` и `index.js`. Пять лет назад я выпустил [Eta](https://github.com/bgub/eta) — уже с гораздо большим набором возможностей, включая тестирование, линтинг, сборку и CI/CD.

Мне казалось, что это уже вполне добротный сетап для разработки, но времена меняются, а экосистема JavaScript движется быстро. Появились новые инструменты, лучшие практики изменились, и правильно публиковать npm-пакеты стало каким-то образом одновременно *и* проще, *и* запутаннее.

Достаточно взглянуть на то, как эволюционировало поле &quot;exports&quot; в `package.json`, если хотите заработать головную боль. Или попробуйте разобраться в правильном сочетании конфигураций TypeScript, бандлеров и CI workflows, чтобы опубликовать библиотеку, которая без проблем работает в Node, Deno, Bun и браузерах. На удивление, сделать это правильно не так-то просто.

Именно поэтому я создал [**ts-base**](https://github.com/bgub/ts-base) — современный стартовый шаблон для библиотеки TypeScript, который берет всю эту сложность на себя. Это продуманное, проверенное в деле решение, спроектированное так, чтобы работать из коробки со всеми основными средами выполнения JavaScript.

## Что такое ts-base?

ts-base — это шаблон библиотеки TypeScript, сочетающий современные инструменты и автоматизированные workflows. Вместо того чтобы начинать с нуля или копировать устаревший шаблонный код, вы получаете полноценную среду разработки с линтингом, тестированием, сборкой, выпуском релизов и публикацией — всё уже настроено и готово к работе.

Шаблон строится на трёх ключевых принципах:

* **Несколько сред выполнения в приоритете**: Бесшовно работает в Node, Deno, Bun и браузерах
* **Автоматизация вместо конфигурации**: Минимум настройки, максимум автоматизации
* **Современный инструментарий**: Только ESM, последняя версия TypeScript и тщательно подобранные зависимости

## Архитектура для разных сред выполнения

Основа ts-base — это дизайн, не привязанный к конкретной среде выполнения. Вместо того чтобы пытаться заставить один файл работать везде (и мучиться с проблемами совместимости), шаблон использует чёткое разделение:

```typescript
// src/internal.ts - Основная логика, без API, специфичных для среды выполнения
export function add(a: number, b: number): number {
  return a + b;
}

export function greet(name: string, options = {}): string {
  const base = `Hello, ${name}`;
  return options.shout ? `${base.toUpperCase()}!` : `${base}.`;
}
```

```typescript
// src/index.ts - адаптер для Node/Bun
export { add, greet } from "./internal";
import { randomBytes } from "node:crypto";

export function getSecureRandomId(): string {
  const timePart = Date.now().toString(36);
  const bytes = randomBytes(12).toString("base64url");
  return `${timePart}-${bytes}`;
}
```

```typescript
// src/browser.ts - Адаптер для браузера
export { add, greet } from "./internal";

export function getSecureRandomId(): string {
  const timePart = Date.now().toString(36);
  const array = new Uint8Array(12);
  crypto.getRandomValues(array);
  const rand = btoa(String.fromCharCode(...array))
    .replaceAll("+", "-")
    .replaceAll("/", "_")
    .replaceAll("=", "");
  return `${timePart}-${rand}`;
}
```

Это даёт чистые импорты для любой среды выполнения:

```typescript
// Node/Bun
import { add, getSecureRandomId } from "@your-package/ts-base";

// Браузер (через бандлер)
import { add, getSecureRandomId } from "@your-package/ts-base/browser";

// Deno (прямые импорты TypeScript)
import {
  add,
  greet,
} from "https://jsr.io/@bgub/ts-base/<version>/src/index.ts";
```

Система сборки использует [tsdown](https://tsdown.dev/) для создания двух оптимизированных сборок: одну для сред Node и отдельную минифицированную сборку для браузеров, обе с картами исходного кода.

## Удобство для разработчика

ts-base сводит ваш инструментарий к нескольким отличным решениям:

**Biome** заменяет и ESLint, и Prettier одним быстрым инструментом. Никаких конфликтов конфигурации или несовместимых плагинов — только единообразное форматирование и линтинг, которые работают из коробки.

**Vitest** обеспечивает молниеносное тестирование со встроенными отчётами о покрытии и настраиваемыми порогами. Тесты выполняются параллельно, имеют нативную поддержку TypeScript и включают полезные возможности вроде моков и снимков.

**Size Limit** автоматически отслеживает размер вашего бандла. Он запускается в CI и оставляет комментарии в пул-реквестах, если ваши изменения увеличивают размер бандла, помогая выявить раздувание до релиза.

Конфигурация TypeScript оптимизирована для современных бандлеров: такие настройки, как `moduleResolution: "bundler"` и `allowImportingTsExtensions: true`, отлично работают с Vite, Rollup и esbuild.

## Автоматизированный CI/CD-пайплайн

Одна из главных сильных сторон ts-base — полностью автоматизированный CI/CD-процесс. Все аспекты контроля качества кода и публикации здесь автоматизированы:

**Контроль качества**: Каждый pull request запускает linting, проверку типов, тесты и отчёты о покрытии. CI загружает данные о покрытии в Codecov и оставляет в PR комментарии с отчётами о влиянии на размер.

![Скриншот запуска CI/CD](/blog-images/ci-run-screenshot.png)

**Управление релизами**: Вместо сложных конфигураций semantic-release ts-base использует Release Please от Google. Когда commit попадает в main, Release Please автоматически открывает «Release PR», обновляет номера версий, генерирует changelog и создаёт теги релиза.

**Автоматизированная публикация**: Когда вы мержите Release PR, GitHub Actions автоматически собирает и публикует пакет сразу в npm и JSR с полной OIDC-проверкой происхождения и аттестацией безопасности.

**Conventional Commits**: Заголовки PR автоматически проходят linting на соответствие формату conventional commit, что обеспечивает единообразную генерацию changelog.

## Почему этот подход работает лучше

Большинство шаблонов для библиотек TypeScript, которые я видел, либо слишком минималистичны (и вам приходится самим разбираться с CI, публикацией и поддержкой нескольких сред выполнения), либо перегружены десятками зависимостей. Я встречал шаблоны с пакетами вроде `@commitlint/cli`, `@commitlint/config-conventional`, `@semantic-release/changelog`, `@semantic-release/git`, `@semantic-release/github`, `@semantic-release/npm` и другими — и всё это только ради публикации через CI!

ts-base идёт другим путём: всего 8 dev-зависимостей. Если выбрать Release Please вместо semantic-release, Biome вместо ESLint+Prettier и Vitest вместо Jest, вы получите более простой граф зависимостей, который легче поддерживать и который с меньшей вероятностью сломается.

Такая философия автоматизации означает меньше конфигурации и меньше мест, где что-то может пойти не так. Release Please берёт на себя повышение версии, генерацию журнала изменений и создание релизов в одном инструменте. Всё остальное делают workflows GitHub Actions.

## Магия Release Please

![Скриншот Release PR от release-please](/blog-images/release-please-pr.png)

Release Please заслуживает отдельного внимания, потому что меняет сам подход к релизам. Вместо того чтобы вручную повышать версии или настраивать сложные пайплайны semantic-release, с Release Please всё работает так:

1. Вы сливаете коммиты в `main`, используя сообщения в формате conventional commit
2. Release Please автоматически открывает или обновляет Release PR с повышением версий и записями в changelog
3. Когда вы готовы к релизу, просто слейте Release PR
4. GitHub Actions автоматически публикует пакет в npm и JSR

Система также поддерживает предварительные релизы. Если вы выпускаете alpha- или beta-версию, она автоматически публикуется в npm под тегом &quot;next&quot;. Вы можете переопределить повышение версии с помощью `Release-As: 2.0.0` в сообщениях коммитов, а также поддерживать несколько веток релизов (например, `2.x` и `3.x`), и для каждой из них будет создаваться свой Release PR.

## Начало работы

Настроить ts-base довольно просто:

1. **Клонируйте и настройте**: Клонируйте репозиторий, удалите папку `.git` и обновите `package.json`, `jsr.json` и `.release-please-manifest.json`, указав данные своего пакета.

2. **Зарезервируйте пакет**: Установите версию `0.0.0` во всех конфигурационных файлах, затем локально выполните `npm publish`, чтобы занять имя пакета в npm.

3. **Настройте публикацию**: В npm настройте пакет так, чтобы 2FA требовалась только для авторизации, но не для публикации, затем добавьте workflow GitHub как доверенного издателя. В JSR создайте пакет и добавьте репозиторий как доверенный источник.

![Скриншот настроек публикации npm](/blog-images/npm-trusted-publisher.png)

4. **Настройте GitHub**: Отправьте код в GitHub, добавьте `CODECOV_TOKEN` в секреты репозитория и настройте правила защиты веток.

5. **Начните разработку**: Добавьте свой код в `src/`, напишите тесты и отправляйте коммиты. Всё остальное сделает Release Please.

Я рекомендую настроить GitHub так, чтобы разрешалось только squash-слияние, и использовать «заголовок pull request и детали коммита» как сообщение коммита по умолчанию. Это помогает поддерживать чистую историю коммитов и соблюдать формат conventional commit.

## Лучшие практики и советы

**Настройки репозитория**: Включите защиту ветки `main` с обязательными проверками статуса. Отключите merge-коммиты, чтобы сохранить линейную историю.

**Точки входа**: Используйте основной экспорт (`@your-package`) для Node/Bun, экспорт для браузера (`@your-package/browser`) для собранного браузерного кода и прямые импорты TypeScript для Deno.

**Кастомизация**: Если вам не нужны отдельные сборки для Node и браузера, удалите неиспользуемую конфигурацию. Этот шаблон рассчитан на то, чтобы вы могли сократить его под свои конкретные задачи.

**Стратегия тестирования**: Шаблон включает примеры тестирования как общего, так и платформозависимого кода, включая мокирование API браузера в тестовой среде Node.
