Восемь лет назад я выпустил свою первую TypeScript-библиотеку с открытым исходным кодом — [Squirrelly](https://github.com/squirrellyjs/squirrelly) — в которой было всего два файла: `package.json` и `index.js`. Пять лет назад я выпустил [Eta](https://github.com/bgub/eta) с куда большим набором возможностей, включая тестирование, линтинг, сборку и CI/CD.

Тогда мне казалось, что это вполне надёжный сетап для разработки, но времена меняются, а экосистема JavaScript развивается стремительно. Появились новые инструменты, лучшие практики изменились, а правильно публиковать npm-пакеты стало каким-то образом одновременно и проще, *и* запутаннее.

Достаточно просто посмотреть на то, как развивалось поле `package.json` «exports», если хотите заработать головную боль. Или попробуйте подобрать правильное сочетание конфигураций TypeScript, сборщиков и CI-процессов, чтобы опубликовать библиотеку, которая безупречно работает в Node, Deno, Bun и браузерах. Сделать это правильно на удивление непросто.

Именно поэтому я создал [**ts-base**](https://github.com/bgub/ts-base) — современный стартовый шаблон для TypeScript-библиотек, который берёт всю эту сложность на себя. В нём уже заложены продуманные решения, он проверен на практике и спроектирован так, чтобы работать из коробки со всеми основными JavaScript-средами выполнения.

## Что такое ts-base?

ts-base — это шаблон для библиотеки на TypeScript, созданный с опорой на современные инструменты и автоматизированные рабочие процессы. Вместо того чтобы начинать с нуля или копировать устаревший шаблонный код, вы получаете полноценную среду разработки с линтингом, тестированием, сборкой, релизами и публикацией — всё уже настроено и готово к работе.

Шаблон строится на трёх ключевых принципах:

* **Поддержка нескольких сред выполнения в приоритете**: Без проблем работает в Node, Deno, Bun и браузерах
* **Автоматизация вместо настройки**: Минимум настройки, максимум автоматизации
* **Современный инструментарий**: Только ESM, новейший TypeScript и тщательно подобранные зависимости

## Архитектура для нескольких сред выполнения

Ключевая особенность ts-base — независимость от конкретной среды выполнения. Вместо того чтобы пытаться заставить один файл работать везде (и бороться с проблемами совместимости), шаблон использует чёткое разделение:

```typescript
// src/internal.ts - Основная логика, без API, специфичных для среды выполнения
export function add(a: number, b: number): number {
  return a + b;
}

export function greet(name: string, options = {}): string {
  const base = `Hello, ${name}`;
  return options.shout ? `${base.toUpperCase()}!` : `${base}.`;
}
```

```typescript
// src/index.ts - адаптер для Node/Bun
export { add, greet } from "./internal";
import { randomBytes } from "node:crypto";

export function getSecureRandomId(): string {
  const timePart = Date.now().toString(36);
  const bytes = randomBytes(12).toString("base64url");
  return `${timePart}-${bytes}`;
}
```

```typescript
// src/browser.ts - адаптер для браузера
export { add, greet } from "./internal";

export function getSecureRandomId(): string {
  const timePart = Date.now().toString(36);
  const array = new Uint8Array(12);
  crypto.getRandomValues(array);
  const rand = btoa(String.fromCharCode(...array))
    .replaceAll("+", "-")
    .replaceAll("/", "_")
    .replaceAll("=", "");
  return `${timePart}-${rand}`;
}
```

Это даёт вам аккуратные импорты для любой среды выполнения:

```typescript
// Node/Bun
import { add, getSecureRandomId } from "@your-package/ts-base";

// Браузер (через сборщик)
import { add, getSecureRandomId } from "@your-package/ts-base/browser";

// Deno (прямой импорт TypeScript)
import {
  add,
  greet,
} from "https://jsr.io/@bgub/ts-base/<version>/src/index.ts";
```

Система сборки использует [tsdown](https://tsdown.dev/) для создания двух оптимизированных бандлов: одного для Node.js и отдельного минифицированного бандла для браузеров, оба с картами исходников.

## Удобство для разработчика

ts-base объединяет ваш инструментарий вокруг нескольких отличных решений:

**Biome** заменяет сразу и ESLint, и Prettier одним быстрым инструментом. Больше никаких конфликтов в конфигурации или несовместимых плагинов — только единообразное форматирование и линтинг, которые работают из коробки.

**Vitest** обеспечивает молниеносное тестирование со встроенными отчётами о покрытии и настраиваемыми пороговыми значениями. Тесты выполняются параллельно, нативно поддерживают TypeScript и включают полезные возможности вроде моков и снапшотов.

**Size Limit** автоматически отслеживает размер вашего бандла. Он запускается в CI и оставляет комментарии в pull request, если ваши изменения увеличивают размер бандла, помогая выявить разрастание ещё до релиза.

Конфигурация TypeScript оптимизирована для современных сборщиков и включает параметры вроде `moduleResolution: "bundler"` и `allowImportingTsExtensions: true`, которые отлично работают с такими инструментами, как Vite, Rollup и esbuild.

## Автоматизированный CI/CD-конвейер

Одна из главных сильных сторон ts-base — полностью автоматизированный CI/CD-процесс. Все аспекты контроля качества кода и публикации выполняются автоматически:

**Проверки качества**: Каждый pull request запускает линтинг, проверку типов, тесты и отчёт о покрытии. CI загружает данные о покрытии в Codecov и оставляет комментарии в PR с отчётами о влиянии на размер.

![Скриншот запуска CI/CD](/blog-images/ci-run-screenshot.png)

**Управление релизами**: Вместо сложных конфигураций semantic-release ts-base использует Release Please от Google. Когда коммиты попадают в main, Release Please автоматически открывает &quot;Release PR&quot;, который обновляет номера версий, генерирует журналы изменений и создаёт теги релизов.

**Автоматизированная публикация**: Когда вы сливаете Release PR, GitHub Actions автоматически собирает и публикует пакет сразу в npm и JSR с полной OIDC-проверкой происхождения и аттестацией безопасности.

**Conventional Commits**: Заголовки PR автоматически проверяются линтером на соответствие формату conventional commits, что обеспечивает единообразную генерацию журналов изменений.

## Почему этот подход лучше

Большинство шаблонов для библиотек TypeScript, которые мне встречались, либо слишком минималистичны (и вам приходится самим разбираться с CI, публикацией и поддержкой нескольких сред выполнения), либо перегружены десятками зависимостей. Я видел шаблоны с пакетами вроде `@commitlint/cli`, `@commitlint/config-conventional`, `@semantic-release/changelog`, `@semantic-release/git`, `@semantic-release/github`, `@semantic-release/npm` и других — и всё это только ради публикации через CI!

ts-base использует другой подход: всего 8 зависимостей для разработки. Если выбрать Release Please вместо semantic-release, Biome вместо ESLint+Prettier и Vitest вместо Jest, получится более простой граф зависимостей, который легче поддерживать и который реже ломается.

Такая философия автоматизации означает меньше конфигурации и меньше мест, где что-то может пойти не так. Release Please берёт на себя повышение версии, генерацию списка изменений и создание релиза в одном инструменте. Всё остальное делают рабочие процессы GitHub Actions.

## Магия Release Please

![Скриншот PR от Release Please](/blog-images/release-please-pr.png)

Release Please заслуживает особого внимания, потому что меняет сам подход к релизам. Вместо того чтобы вручную повышать версии или настраивать сложные пайплайны semantic-release, Release Please работает так:

1. Вы сливаете коммиты в `main`, используя сообщения коммитов по Conventional Commits
2. Release Please автоматически открывает или обновляет «Release PR» с повышением версий и записями в журнале изменений
3. Когда вы готовы к релизу, просто слейте Release PR
4. GitHub Actions автоматически публикует пакет в npm и JSR

Система также поддерживает предварительные релизы. Если вы выпускаете alpha- или beta-версию, она автоматически публикуется под тегом `next` в npm. Вы можете переопределить повышение версии с помощью `Release-As: 2.0.0` в сообщениях коммитов, а также поддерживать несколько релизных веток (например, `2.x` и `3.x`), каждая из которых будет получать свой собственный Release PR.

## Начало работы

Настроить ts-base довольно просто:

1. **Клонируйте и настройте**: Клонируйте репозиторий, удалите папку `.git` и обновите `package.json`, `jsr.json` и `.release-please-manifest.json`, указав данные своего пакета.

2. **Зарезервируйте пакет**: Установите версию `0.0.0` во всех конфигурационных файлах, затем локально выполните `npm publish`, чтобы занять имя пакета в npm.

3. **Настройте публикацию**: В npm настройте пакет так, чтобы 2FA требовалась только для авторизации, а не для публикации, затем добавьте workflow GitHub как доверенного издателя. В JSR создайте пакет и добавьте репозиторий как доверенный источник.

![Скриншот настроек публикации npm](/blog-images/npm-trusted-publisher.png)

4. **Настройте GitHub**: Отправьте код в GitHub, добавьте `CODECOV_TOKEN` в секреты репозитория и настройте правила защиты веток.

5. **Начните разработку**: Добавьте код в `src/`, напишите тесты и отправляйте коммиты. Обо всём остальном позаботится Release Please.

Я рекомендую настроить GitHub так, чтобы разрешалось только squash-слияние, а в качестве сообщения коммита по умолчанию использовать «заголовок pull request и детали коммита». Это помогает поддерживать чистую историю коммитов и соблюдать conventional commits.

## Лучшие практики и советы

**Настройки репозитория**: Включите защиту ветки `main` с обязательными проверками статуса. Отключите merge-коммиты, чтобы история оставалась линейной.

**Точки входа**: Используйте основной экспорт (`@your-package`) для Node/Bun, браузерный экспорт (`@your-package/browser`) для браузерного кода в сборке и прямой импорт TypeScript для Deno.

**Настройка**: Если вам не нужны отдельные сборки для Node и браузера, удалите лишнюю конфигурацию. Шаблон рассчитан на то, чтобы его можно было легко адаптировать под ваши конкретные задачи.

**Стратегия тестирования**: Шаблон включает примеры тестирования как общего, так и платформенно-специфичного кода, включая мокирование браузерных API в тестовой среде Node.
