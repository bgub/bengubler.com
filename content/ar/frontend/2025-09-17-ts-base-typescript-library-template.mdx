قبل ثماني سنوات، أطلقت أول مكتبة TypeScript مفتوحة المصدر لي — [Squirrelly](https://github.com/squirrellyjs/squirrelly) — وكانت تتكوّن من ملفين فقط: `package.json` و`index.js`. وقبل خمس سنوات، أطلقت [Eta](https://github.com/bgub/eta) بميزات أكثر بكثير، منها الاختبارات، وفحص الشيفرة، وbundling، وCI/CD.

كنت أظن أن ذلك إعداد تطوير قوي جدًا، لكن الزمن يتغير ومنظومة JavaScript تتطور بسرعة. ظهرت أدوات جديدة، وتطورت أفضل الممارسات، وأصبح تعقيد نشر حزمة npm بالشكل الصحيح أسهل *وفي الوقت نفسه* أكثر إرباكًا، بطريقة ما.

يكفي أن تلقي نظرة على تطور حقل &quot;exports&quot; في `package.json` إذا كنت تريد صداعًا. أو جرّب أن تكتشف التوليفة المناسبة من إعدادات TypeScript، وأدوات bundling، ومسارات عمل CI لنشر مكتبة تعمل بسلاسة على Node وDeno وBun والمتصفحات. والمفاجئ أن إتقان ذلك أصعب مما يبدو.

ولهذا أنشأت [**ts-base**](https://github.com/bgub/ts-base) — قالبًا حديثًا للبدء في تطوير مكتبات TypeScript يتكفّل بكل هذا التعقيد نيابةً عنك. إنه ذو توجهات واضحة، ومجرّب عمليًا، ومصمم ليعمل مباشرةً ومن دون إعداد إضافي مع جميع بيئات تشغيل JavaScript الرئيسية.

## ما هو ts-base؟

ts-base هو قالب لمكتبة TypeScript يعتمد أدوات حديثة وسير عمل مؤتمتة. فبدلًا من البدء من الصفر أو نسخ قوالب قديمة، تحصل على بيئة تطوير متكاملة تشمل فحص الشيفرة، والاختبار، والبناء، والإصدار، والنشر — وكلها مُعدّة مسبقًا وجاهزة للاستخدام.

يقوم القالب على ثلاثة مبادئ أساسية:

* **تعدد البيئات التشغيلية أولًا**: يعمل بسلاسة على Node وDeno وBun وفي المتصفحات
* **الأتمتة بدلًا من الإعداد**: أقل قدر من الإعداد، وأقصى قدر من الأتمتة
* **أدوات حديثة**: ‏ESM فقط، وأحدث TypeScript، وتبعيات مختارة بعناية

## بنية متعددة لبيئات التشغيل

يكمن جوهر ts-base في تصميمه غير المرتبط ببيئة تشغيل معيّنة. فبدلًا من محاولة جعل ملف واحد يعمل في كل مكان (وما يرافق ذلك من مشكلات التوافق المزعجة)، يستخدم القالب فصلًا واضحًا:

```typescript
// src/internal.ts - المنطق الأساسي، بدون واجهات برمجية خاصة ببيئة التشغيل
export function add(a: number, b: number): number {
  return a + b;
}

export function greet(name: string, options = {}): string {
  const base = `Hello, ${name}`;
  return options.shout ? `${base.toUpperCase()}!` : `${base}.`;
}
```

```typescript
// src/index.ts - محوّل Node/Bun
export { add, greet } from "./internal";
import { randomBytes } from "node:crypto";

export function getSecureRandomId(): string {
  const timePart = Date.now().toString(36);
  const bytes = randomBytes(12).toString("base64url");
  return `${timePart}-${bytes}`;
}
```

```typescript
// src/browser.ts - محوّل المتصفح
export { add, greet } from "./internal";

export function getSecureRandomId(): string {
  const timePart = Date.now().toString(36);
  const array = new Uint8Array(12);
  crypto.getRandomValues(array);
  const rand = btoa(String.fromCharCode(...array))
    .replaceAll("+", "-")
    .replaceAll("/", "_")
    .replaceAll("=", "");
  return `${timePart}-${rand}`;
}
```

يتيح لك هذا استيرادات نظيفة في كل بيئة تشغيل:

```typescript
// Node/Bun
import { add, getSecureRandomId } from "@your-package/ts-base";

// المتصفح (عبر أداة التجميع)
import { add, getSecureRandomId } from "@your-package/ts-base/browser";

// Deno (استيراد TypeScript مباشرة)
import {
  add,
  greet,
} from "https://jsr.io/@bgub/ts-base/<version>/src/index.ts";
```

يستخدم نظام البناء [tsdown](https://tsdown.dev/) لإنشاء حزمتين محسّنتين: واحدة لبيئات Node وأخرى منفصلة ومصغّرة للمتصفحات، وكلتاهما تتضمنان خرائط مصدر.

## تجربة المطوّر

يجمع ts-base أدواتك ضمن مجموعة صغيرة من الخيارات الممتازة:

**Biome** يحلّ محل كلٍّ من ESLint وPrettier بأداة واحدة سريعة. لا مزيد من تعارضات الإعدادات أو مشاكل توافق الإضافات — فقط تنسيق وفحص متّسقان يعملان مباشرة دون عناء.

**Vitest** يوفّر اختبارات فائقة السرعة مع تقارير تغطية مدمجة وحدود قابلة للتخصيص. تُشغَّل الاختبارات بالتوازي، وتدعم TypeScript دعمًا أصيلًا، وتتضمن ميزات مفيدة مثل المحاكاة واللقطات.

**Size Limit** يراقب حجم الحزمة تلقائيًا. يعمل ضمن CI ويضيف تعليقات إلى طلبات السحب عندما تؤدي تغييراتك إلى زيادة حجم الحزمة، مما يساعدك على اكتشاف التضخّم قبل وصوله إلى الإصدار.

إعدادات TypeScript مُحسّنة لحزم البناء الحديثة، مع خيارات مثل `moduleResolution: "bundler"` و`allowImportingTsExtensions: true` التي تعمل بكفاءة عالية مع أدوات مثل Vite وRollup وesbuild.

## خط أنابيب CI/CD المؤتمت

من أبرز نقاط قوة ts-base إعدادُه المتكامل بالكامل لـ CI/CD. فجميع جوانب جودة الشيفرة والنشر مؤتمتة:

**بوابات الجودة**: يؤدي كل طلب سحب إلى تشغيل فحص الشيفرة، والتحقق من الأنواع، والاختبارات، وإعداد تقارير التغطية. كما يرفع CI بيانات التغطية إلى Codecov ويضيف تعليقات على طلبات السحب تتضمن تقارير عن تأثيرها على الحجم.

![لقطة شاشة لتشغيل CI/CD](/blog-images/ci-run-screenshot.png)

**إدارة الإصدارات**: بدلًا من إعدادات semantic-release المعقدة، يستخدم ts-base أداة Release Please من Google. وعند وصول الالتزامات إلى الفرع main، تفتح Release Please تلقائيًا &quot;Release PR&quot; لتحديث أرقام الإصدارات، وإنشاء سجلات التغيير، وإنشاء وسوم الإصدار.

**النشر المؤتمت**: عند دمج Release PR، تتولى GitHub Actions تلقائيًا بناء حزمتك ونشرها إلى كلٍّ من npm وJSR مع إثبات منشأ كامل عبر OIDC وشهادة اعتماد أمنية.

**Conventional Commits**: تُفحَص عناوين طلبات السحب تلقائيًا للتأكد من التزامها بتنسيق conventional commit، مما يضمن إنشاء سجلات تغيير متسقة.

## لماذا ينجح هذا النهج بصورة أفضل

معظم قوالب مكتبات TypeScript التي رأيتها تكون إما شديدة البساطة (فتترك لك مهمة التعامل مع CI والنشر ودعم بيئات تشغيل متعددة) أو شديدة التعقيد مع عشرات التبعيات. وقد رأيت قوالب تتضمن حزمًا مثل `@commitlint/cli` و`@commitlint/config-conventional` و`@semantic-release/changelog` و`@semantic-release/git` و`@semantic-release/github` و`@semantic-release/npm` وغيرها، وكل ذلك فقط من أجل النشر عبر CI!

أما ts-base فيتبنّى نهجًا مختلفًا، إذ يكتفي بإجمالي 8 تبعيات تطوير فقط. وباختيار Release Please بدلًا من semantic-release، وBiome بدلًا من ESLint+Prettier، وVitest بدلًا من Jest، تحصل على شجرة تبعيات أبسط، أسهل صيانةً، وأقل عرضة للتعطل.

وتعني فلسفة الأتمتة إعدادات أقل ونقاطًا أقل يمكن أن تحدث فيها المشكلات. إذ يتولى Release Please رفع رقم الإصدار، وإنشاء سجل التغييرات، وإصدار النسخة الجديدة ضمن أداة واحدة. أما مسارات عمل GitHub Actions فتتولى كل ما تبقى.

## سحر Release Please

![لقطة شاشة لطلب السحب الخاص بـ Release Please](/blog-images/release-please-pr.png)

يستحق Release Please اهتمامًا خاصًا لأنه يغيّر طريقة تفكيرك في الإصدارات. فبدلًا من زيادة أرقام الإصدارات يدويًا أو إعداد مسارات semantic-release معقدة، يعمل Release Please كالتالي:

1. تدمج الـ commits في `main` باستخدام رسائل Conventional Commits القياسية
2. يفتح Release Please تلقائيًا &quot;Release PR&quot; أو يحدّثه، مع زيادة أرقام الإصدارات وإضافة إدخالات إلى سجل التغييرات
3. عندما تصبح جاهزًا لإصدار نسخة جديدة، ما عليك سوى دمج Release PR
4. تتولى GitHub Actions النشر تلقائيًا إلى npm وJSR

يدعم النظام أيضًا الإصدارات التمهيدية. فإذا أصدرت نسخة alpha أو beta، فسيُنشر تلقائيًا تحت الوسم &quot;next&quot; على npm. ويمكنك أيضًا تجاوز زيادة رقم الإصدار باستخدام `Release-As: 2.0.0` في رسائل commit، كما يمكنك الاحتفاظ بعدة فروع للإصدار (مثل `2.x` و`3.x`) بحيث يحصل كلٌّ منها على Release PR خاص به.

## البدء

إعداد ts-base سهل ومباشر:

1. **الاستنساخ والتخصيص**: انسخ المستودع، واحذف مجلد `.git`، ثم حدّث ملفات `package.json` و`jsr.json` و`.release-please-manifest.json` بتفاصيل حزمتك.

2. **احجز اسم حزمتك**: اضبط الإصدار على `0.0.0` في جميع ملفات الإعداد، ثم شغّل `npm publish` محليًا لحجز اسم حزمتك على npm.

3. **إعداد النشر**: في npm، اضبط حزمتك بحيث تتطلب المصادقة الثنائية (2FA) للتفويض فقط (وليس للنشر)، ثم أضف سير عمل GitHub كناشر موثوق. وفي JSR، أنشئ حزمتك وأضف المستودع كمصدر موثوق.

![لقطة شاشة لإعدادات النشر في npm](/blog-images/npm-trusted-publisher.png)

4. **إعداد GitHub**: ارفع مشروعك إلى GitHub، وأضف `CODECOV_TOKEN` كسرّ للمستودع، ثم اضبط قواعد حماية الفروع.

5. **ابدأ التطوير**: أضف شيفرتك إلى `src/`، واكتب الاختبارات، وادفع التغييرات. سيتولى Release Please الباقي.

أوصي بإعداد GitHub بحيث يسمح فقط بدمج squash، واستخدام &quot;عنوان طلب السحب وتفاصيل الالتزام&quot; كرسالة الالتزام الافتراضية. يساعد ذلك في الحفاظ على سجل التزامات نظيف ويضمن الالتزام بمعيار Conventional Commits.

## أفضل الممارسات والنصائح

**إعدادات المستودع**: فعِّل حماية الفروع على `main` مع اشتراط اجتياز فحوصات الحالة المطلوبة. وعطِّل عمليات الدمج من نوع `merge commits` للحفاظ على سجلٍّ خطي.

**نقاط الدخول**: استخدم التصدير الرئيسي (`@your-package`) مع Node/Bun، وتصدير المتصفح (`@your-package/browser`) لشفرة المتصفح المجمّعة، وعمليات استيراد TypeScript المباشرة مع Deno.

**التخصيص**: إذا لم تكن بحاجة إلى بنيات منفصلة لـ Node والمتصفح، فاحذف الإعدادات غير المستخدمة. صُمِّم هذا القالب بحيث يمكن تقليصه ليلائم احتياجاتك الفعلية.

**استراتيجية الاختبار**: يتضمن القالب أمثلة على اختبار كلٍّ من الشفرة المشتركة والشفرة الخاصة بكل منصة، بما في ذلك محاكاة واجهات برمجة تطبيقات المتصفح ضمن بيئة اختبار Node.

## ختامًا

لا ينبغي أن يتطلّب نشر مكتبة TypeScript دكتوراه في إعداد أدوات التطوير. يوفّر لك ts-base أساسًا حديثًا وعمليًا يتكفّل بالتعقيد، لتتمكّن من التركيز على بناء برمجيات رائعة.

يجسّد هذا القالب خلاصة ثماني سنوات من الدروس المستفادة من صيانة مشاريع مفتوحة المصدر. هل أنت مستعد لتجربته؟ ألقِ نظرة على [مستودع ts-base](https://github.com/bgub/ts-base) وابدأ ببناء مكتبتك التالية.
