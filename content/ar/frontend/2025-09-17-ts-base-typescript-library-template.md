قبل ثماني سنوات، أطلقتُ أول مكتبة TypeScript مفتوحة المصدر لي — [Squirrelly](https://github.com/squirrellyjs/squirrelly) — وكانت تضم ملفين فقط: `package.json` و`index.js`. وقبل خمس سنوات، أطلقتُ [Eta](https://github.com/bgub/eta) بمزايا أكثر بكثير، منها الاختبارات، وفحص الشيفرة، والتجميع، وCI/CD.

كنت أظن أن هذه كانت بيئة تطوير قوية جدًا، لكن الزمن يتغير ومنظومة JavaScript تتحرك بسرعة. ظهرت أدوات جديدة، وتطورت أفضل الممارسات، وأصبح نشر حزمة npm على النحو الصحيح أسهل من جهة *وفي الوقت نفسه* أكثر إرباكًا من جهة أخرى.

يكفي أن تنظر إلى تطور حقل &quot;exports&quot; في `package.json` إذا كنت تبحث عن صداع. أو جرّب معرفة التوليفة المناسبة من إعدادات TypeScript، وأدوات التجميع، وسير عمل CI لنشر مكتبة تعمل بسلاسة عبر Node وDeno وBun والمتصفحات. ومن المدهش فعلًا كم يصعب ضبط ذلك كما ينبغي.

لهذا أنشأتُ [**ts-base**](https://github.com/bgub/ts-base) — قالب بداية حديثًا لمكتبات TypeScript يتولى كل هذا التعقيد نيابةً عنك. إنه مبني على اختيارات واضحة، ومجرّب عمليًا، ومصمم ليعمل مباشرةً دون إعدادات إضافية مع جميع بيئات تشغيل JavaScript الرئيسية.

## ما هو ts-base؟

ts-base هو قالب لمكتبة TypeScript يعتمد على أدوات حديثة وسير عمل مؤتمتة. فبدلًا من البدء من الصفر أو نسخ قوالب قديمة، تحصل على بيئة تطوير متكاملة تشمل فحص الشيفرة والاختبار والبناء والإصدار والنشر — وكل ذلك مُعدّ مسبقًا وجاهز للاستخدام.

يقوم القالب على ثلاثة مبادئ أساسية:

* **مصمم لعدة بيئات تشغيل أولًا**: يعمل بسلاسة على Node وDeno وBun وفي المتصفحات
* **الأتمتة بدلًا من الإعداد**: أقل قدر من الإعداد، وأكبر قدر من الأتمتة
* **أدوات حديثة**: ESM فقط، وأحدث إصدار من TypeScript، وتبعيات مختارة بعناية

## البنية متعددة بيئات التشغيل

يكمن جوهر ts-base في تصميمه المستقل عن بيئة التشغيل. فبدلًا من محاولة جعل ملف واحد يعمل في كل مكان (وما يرافق ذلك من مشكلات توافق مزعجة)، يستخدم القالب فصلًا واضحًا ونظيفًا:

```typescript
// src/internal.ts - المنطق الأساسي، بدون واجهات برمجة التطبيقات الخاصة ببيئة التشغيل
export function add(a: number, b: number): number {
  return a + b;
}

export function greet(name: string, options = {}): string {
  const base = `Hello, ${name}`;
  return options.shout ? `${base.toUpperCase()}!` : `${base}.`;
}
```

```typescript
// src/index.ts - محوّل Node/Bun
export { add, greet } from "./internal";
import { randomBytes } from "node:crypto";

export function getSecureRandomId(): string {
  const timePart = Date.now().toString(36);
  const bytes = randomBytes(12).toString("base64url");
  return `${timePart}-${bytes}`;
}
```

```typescript
// src/browser.ts - محوّل المتصفح
export { add, greet } from "./internal";

export function getSecureRandomId(): string {
  const timePart = Date.now().toString(36);
  const array = new Uint8Array(12);
  crypto.getRandomValues(array);
  const rand = btoa(String.fromCharCode(...array))
    .replaceAll("+", "-")
    .replaceAll("/", "_")
    .replaceAll("=", "");
  return `${timePart}-${rand}`;
}
```

يوفّر لك هذا استيرادات نظيفة لكل بيئة تشغيل:

```typescript
// Node/Bun
import { add, getSecureRandomId } from "@your-package/ts-base";

// المتصفح (عبر أداة التجميع)
import { add, getSecureRandomId } from "@your-package/ts-base/browser";

// Deno (استيراد TypeScript مباشرةً)
import {
  add,
  greet,
} from "https://jsr.io/@bgub/ts-base/<version>/src/index.ts";
```

يستخدم نظام البناء [tsdown](https://tsdown.dev/) لإنشاء حزمتين محسّنتين: واحدة لبيئات Node، وأخرى مصغّرة منفصلة للمتصفحات، وكلتاهما تتضمنان خرائط مصدر.

## تجربة المطور

يجمع ts-base أدواتك ضمن مجموعة صغيرة من الخيارات الممتازة:

**Biome** يحل محل كلٍّ من ESLint وPrettier بأداة واحدة سريعة. لا مزيد من تعارضات الإعدادات أو مشاكل توافق الإضافات — فقط تنسيق متسق وفحص للشيفرة يعمل مباشرةً دون إعداد إضافي.

**Vitest** يوفّر اختبارات فائقة السرعة مع تقارير تغطية مدمجة وحدود قابلة للتخصيص. تعمل الاختبارات بالتوازي، وتدعم TypeScript دعمًا أصيلًا، وتتضمن ميزات مفيدة مثل المحاكاة واللقطات.

**Size Limit** يراقب حجم الحزمة تلقائيًا. يعمل ضمن CI ويضيف تعليقات على طلبات السحب عندما تؤدي تغييراتك إلى زيادة حجم الحزمة، مما يساعدك على اكتشاف التضخم قبل إطلاقه.

إعدادات TypeScript مُحسّنة لأدوات التجميع الحديثة، مع خيارات مثل `moduleResolution: "bundler"` و`allowImportingTsExtensions: true` التي تعمل بكفاءة عالية مع أدوات مثل Vite وRollup وesbuild.

## مسار CI/CD مؤتمت

من أكبر نقاط قوة ts-base إعدادُه المتكامل لـ CI/CD. فكل جانب من جوانب جودة الشيفرة والنشر مؤتمت:

**بوابات الجودة**: يفعّل كل طلب سحب تلقائيًا فحص الشيفرة، والتحقق من الأنواع، والاختبارات، وتقارير التغطية. ويرفع CI بيانات التغطية إلى Codecov ويضيف تعليقات على طلبات السحب تتضمن تقارير عن أثر الحجم.

![لقطة شاشة لتشغيل CI/CD](/blog-images/ci-run-screenshot.png)

**إدارة الإصدارات**: بدلًا من إعدادات semantic-release المعقدة، يستخدم ts-base أداة Release Please من Google. وعندما تصل عمليات commit إلى الفرع main، تفتح Release Please تلقائيًا &quot;طلب سحب للإصدار&quot; يحدّث أرقام الإصدارات، ويولّد سجلات التغييرات، وينشئ وسوم الإصدارات.

**النشر المؤتمت**: عند دمج طلب السحب الخاص بالإصدار، تتولى GitHub Actions تلقائيًا بناء حزمتك ونشرها إلى كلٍّ من npm وJSR مع إثبات OIDC كامل وشهادة اعتماد أمنية.

**Conventional Commits**: تُفحَص عناوين طلبات السحب تلقائيًا للتأكد من التزامها بتنسيق Conventional Commits، مما يضمن توليد سجلات تغييرات متسقة.

## لماذا ينجح هذا النهج بصورة أفضل

معظم قوالب مكتبات TypeScript التي رأيتها إما أنها بسيطة أكثر من اللازم (فتترك لك مهمة تدبر CI والنشر ودعم التشغيل عبر بيئات متعددة) أو معقدة أكثر مما ينبغي، مع عشرات الاعتماديات. وقد رأيت قوالب تتضمن حزمًا مثل `@commitlint/cli`, `@commitlint/config-conventional`, `@semantic-release/changelog`, `@semantic-release/git`, `@semantic-release/github`, `@semantic-release/npm` وغيرها، وكل ذلك فقط لأجل النشر عبر CI!

يتبع ts-base نهجًا مختلفًا، إذ يكتفي بـ 8 اعتماديات تطوير فقط. وباختيار Release Please بدلًا من semantic-release، وBiome بدلًا من ESLint+Prettier، وVitest بدلًا من Jest، تحصل على شجرة اعتماديات أبسط، أسهل في الصيانة، وأقل عرضة للتعطل.

وتعني فلسفة الأتمتة هذه إعدادات أقل ونقاطًا أقل يمكن أن تحدث فيها المشاكل. إذ يتولى Release Please رفع الإصدار وإنشاء سجل التغييرات وإصدار النسخة الجديدة ضمن أداة واحدة. أما سير العمل في GitHub Actions فيتولى كل شيء آخر.

## سحر Release Please

![لقطة شاشة لطلب سحب release-please](/blog-images/release-please-pr.png)

يستحق Release Please اهتمامًا خاصًا لأنه يغيّر الطريقة التي تفكّر بها في الإصدارات. فبدلًا من زيادة أرقام الإصدارات يدويًا أو إعداد مسارات semantic-release معقّدة، يعمل Release Please كالتالي:

1. تدمج عمليات commit في `main` باستخدام رسائل Conventional Commits
2. يفتح Release Please تلقائيًا &quot;طلب سحب للإصدار&quot; أو يحدّثه متضمّنًا زيادة أرقام الإصدارات وإدخالات سجل التغييرات
3. عندما تكون مستعدًا للإصدار، ما عليك سوى دمج طلب السحب للإصدار
4. تنشر GitHub Actions الحزمة تلقائيًا على npm و JSR

ويدعم النظام أيضًا الإصدارات التمهيدية. فإذا أصدرت نسخة alpha أو beta، فسيُنشر تلقائيًا تحت الوسم &quot;next&quot; على npm. ويمكنك أيضًا تجاوز زيادة رقم الإصدار باستخدام `Release-As: 2.0.0` في رسائل commit، كما يمكنك الاحتفاظ بعدة فروع للإصدار (مثل `2.x` و `3.x`) بحيث يحصل كل فرع منها على طلب سحب خاص بالإصدار.

## البدء

إعداد ts-base سهل ومباشر:

1. **استنسخ وخصّص**: استنسخ المستودع، واحذف مجلد `.git`، وحدّث `package.json` و`jsr.json` و`.release-please-manifest.json` ببيانات حزمتك.

2. **احجز حزمتك**: اضبط الإصدار على `0.0.0` في جميع ملفات الإعداد، ثم شغّل `npm publish` محليًا لحجز اسم حزمتك على npm.

3. **اضبط النشر**: في npm، اضبط حزمتك بحيث تتطلب المصادقة الثنائية 2FA للتفويض فقط (وليس للنشر)، ثم أضف سير العمل الخاص بك على GitHub كناشر موثوق. وفي JSR، أنشئ حزمتك وأضف المستودع كمصدر موثوق.

![لقطة شاشة لإعدادات النشر في npm](/blog-images/npm-trusted-publisher.png)

4. **أعِدّ GitHub**: ادفع التغييرات إلى GitHub، وأضف `CODECOV_TOKEN` إلى أسرار المستودع، واضبط قواعد حماية الفروع.

5. **ابدأ التطوير**: أضف شيفرتك إلى `src/`، واكتب الاختبارات، وادفع الـcommits. وسيتولى Release Please الباقي.

أوصي بضبط GitHub بحيث يسمح فقط بالدمج من نوع squash، واستخدام «عنوان طلب السحب وتفاصيل الـcommit» كرسالة commit افتراضية. فهذا يحافظ على نظافة سجل الـcommit ويضمن الالتزام بـConventional Commits.

## أفضل الممارسات والنصائح

**إعدادات المستودع**: فعّل حماية الفروع على `main` مع فحوصات الحالة المطلوبة. وعطّل merge commits للحفاظ على سجل خطي.

**نقاط الدخول**: استخدم التصدير الرئيسي (`@your-package`) لـ Node/Bun، وتصدير المتصفح (`@your-package/browser`) للكود المجمّع للمتصفح، وعمليات استيراد TypeScript المباشرة لـ Deno.

**التخصيص**: إذا لم تكن بحاجة إلى إصدارات منفصلة لـ Node والمتصفح، فاحذف الإعدادات غير المستخدمة. صُمّم هذا القالب بحيث يمكن تقليصه ليلائم احتياجاتك الفعلية.

**استراتيجية الاختبار**: يتضمن القالب أمثلة على اختبار كلٍّ من الكود المشترك والكود الخاص بكل منصة، بما في ذلك محاكاة واجهات برمجة التطبيقات الخاصة بالمتصفح في بيئة اختبار Node.

## ختامًا

لا ينبغي أن يتطلب نشر مكتبة TypeScript دكتوراه في تهيئة الأدوات. يمنحك ts-base أساسًا حديثًا بنهج واضح يتكفّل بالتعقيد، لتتمكن من التركيز على بناء برمجيات رائعة.

يجسّد هذا القالب خلاصة ثماني سنوات من الدروس المستفادة من صيانة مشاريع مفتوحة المصدر. هل أنت مستعد لتجربته؟ ألقِ نظرة على [مستودع ts-base](https://github.com/bgub/ts-base) وابدأ في بناء مكتبتك التالية.
