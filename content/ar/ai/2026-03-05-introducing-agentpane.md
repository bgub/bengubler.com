# التعرّف على agentpane

agentpane هو واجهة ويب محلية لوكلاء البرمجة بالذكاء الاصطناعي. تشغّل `npx agentpane`، فيُفتح في متصفحك، ويمكنك التحدث إلى Claude Code أو Codex أو أي وكيل يدعم [ACP (Agent Client Protocol)](https://github.com/anthropics/agent-protocol) — وكل ذلك من خلال واجهة متعددة اللوحات مع البث اللحظي والتخزين الدائم وإدارة الجلسات:

```bash
npx agentpane
# → http://localhost:6767
```

اختر وكيلاً، واختر مجلد عمل، ثم ابدأ جلسة. يعمل الوكيل كعملية فرعية على جهازك. من دون سحابة، ولا نشر، ولا حسابات.

![شاشة إعداد agentpane مع اختيار الوكيل والمجلد](/blog-images/agentpane-setup.png)

## متعدد اللوحات، متعدد الوكلاء

أهم ما أردته هو القدرة على تشغيل عدة وكلاء في الوقت نفسه ورؤيتهم جنبًا إلى جنب. يوفّر لك agentpane ما يصل إلى أربع لوحات قابلة لتغيير الحجم مع علامات تبويب. يمكنك سحب الجلسات من الشريط الجانبي إلى أي لوحة، أو سحب علامة تبويب لتقسيمها إلى لوحة جديدة. ويبقى التخطيط محفوظًا حتى بعد إعادة التحديث.

وهذا يعني أنه يمكنك جعل Claude Code يعمل على الواجهة الخلفية في لوحة، بينما يتولى Codex إعادة هيكلة الواجهة الأمامية في لوحة أخرى — على الشاشة نفسها، وفي التطبيق نفسه.

![جلستان تعملان جنبًا إلى جنب — Codex على اليسار، وClaude Code على اليمين](/blog-images/agentpane-hero.png)

## البنية

يتكوّن التطبيق من عمليتين:

* **خادم API** (Hono، المنفذ 3456) — يشغّل الوكلاء، ويدير الجلسات، ويتولى التخزين الدائم
* **الواجهة الأمامية للويب** (Next.js، المنفذ 6767) — واجهة المستخدم، مع إعادة توجيه طلبات `/api` إلى الواجهة الخلفية

الواجهة الخلفية مبنية باستخدام [Effect.ts](https://effect.website/) وتستخدم حقن التبعيات عبر طبقات خدمات قابلة للتركيب:

```
SessionRepo (DB)
  → ConnectionManager (agent subprocess lifecycle)
  → PromptEngine (turn orchestration)
  → EventHub / EventBroadcaster (SSE)
  → WriteQueue (batched DB writes)
```

يتواصل الوكلاء عبر ACP — أي JSON-RPC 2.0 عبر stdio. عندما ترسل موجّهًا، ينشئ `PromptEngine` سجلات التبادلات، ويمرّر الرسالة إلى العملية الفرعية للوكيل عبر ACP، ثم يُعيد بثّ النتائج عبر Server-Sent Events.

### البث مع أمان إعادة الاتصال

تشترك الواجهة الأمامية في نقطة نهاية SSE لكل جلسة. ويُخصَّص لكل حدث معرّف تصاعدي. ويحتفظ `EventBroadcaster` بمخزن دائري (512KB / 1000 حدث)، بحيث إذا انقطع اتصال المتصفح ثم أعاد الاتصال، يعيد إرسال الأحداث الفائتة باستخدام ترويسة `Last-Event-ID`. لا فجوات في المحادثة، ولا حاجة إلى تحديث الصفحة يدويًا.

### تخزين دائم آمن عند التعطل

تُحفَظ الحالة بالكامل في قاعدة بيانات SQLite محلية في `~/.agentpane/agentpane.db`. لكن عمليات الكتابة لا تحدث فورًا — إذ تقوم `WriteQueue` بتجميع العمليات ثم تُفرِّغها على دفعات كل 50 مللي ثانية. وقبل التفريغ، تُحفَظ العمليات في جدول للتعافي. وإذا تعطّل الخادم أثناء التفريغ، فإنه يستأنف من حيث توقّف عند إعادة التشغيل. لا تُفقَد أي رسائل.

```sql
-- جدول الاسترداد
CREATE TABLE write_queue_ops (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  op_json TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## ما الذي يراه الوكلاء

لا يهم agentpane أي نموذج يشغّل الوكيل — فهو يتواصل عبر ACP. عند الاتصال بجلسة، يقوم `ConnectionManager` بما يلي:

1. يحدّد الملف التنفيذي للوكيل (يفحص `node_modules/.bin/` للتوافق مع npm)
2. يشغّله كعملية فرعية باستخدام قنوات stdio
3. يتفاوض على إمكانات ACP (المصادقة، الإعدادات، الأوضاع، الأوامر)
4. يحاول استئناف الجلسة عند إعادة الاتصال بجلسة سابقة

تتكيّف واجهة المستخدم مع كل ما يدعمه الوكيل. إذا كان الوكيل يوفّر خيارات إعداد، فستظهر كقوائم منسدلة. وإذا كان يدعم أوامر `/`، فستحصل على إكمال تلقائي. وإذا كان يبث كتل التفكير، فستُعرض كأقسام قابلة للطي.

## الواجهة الأمامية

تطبيق الويب مبنيّ باستخدام Next.js 16 مع React 19 وReact Compiler — من دون أي استخدام يدوي لـ `useMemo` أو `useCallback` في أي موضع. وتنقسم الحالة إلى طبقتين من السياق:

* **SessionProvider** — الجلسة النشطة، فحوصات الصحة، عمليات CRUD الخاصة بالجلسة
* **LayoutProvider** — إعدادات اللوحات، ترتيب علامات التبويب، السحب والإفلات

تُدار حالة الخادم (الجلسات، المحادثات، استهلاك التوكنات) باستخدام TanStack React Query، وتُبطَل صلاحيتها عند ورود أحداث SSE. ويُعرَض Markdown أثناء تدفّقه باستخدام [Streamdown](https://github.com/anthropics/streamdown) مع تمييز الشيفرة عبر Shiki.
