# مقدمة إلى helm

helm هو إطار عمل TypeScript مضبوط الأنواع لوكلاء الذكاء الاصطناعي. بدلًا من تشغيل أوامر الصدفة وتحليل السلاسل النصية، تستدعي الوكلاء دوالًا مضبوطة الأنواع ذات مدخلات ومخرجات منظَّمة:

```ts
import { createHelm, git, fs, grep } from "@bgub/helm";

const agent = createHelm({
  permissions: {
    "fs.read": "allow",
    "fs.write": "ask",
    "fs.remove": "deny",
    "git.status": "allow",
    "git.*": "ask",
  },
  onPermissionRequest: async (operation, args) => {
    return confirm(`Allow ${operation}?`);
  },
})
  .use(fs())
  .use(git())
  .use(grep());

const { staged, unstaged, branch } = await agent.git.status();
const { content } = await agent.fs.read("./package.json");
```

تسجّل &quot;المهارات&quot; — وهي مجموعات من العمليات المرتبطة — باستخدام نمط البناء. ويستنتج TypeScript النوع الكامل في كل خطوة. ولكل عملية مستوى أذونات: `allow` (تعمل فورًا)، أو `ask` (تتوقف بانتظار الموافقة)، أو `deny` (تُطلق `PermissionDeniedError`). وتُحسم الأذونات بحسب ترتيب الأولوية: تطابق تام → wildcard → الإعداد الافتراضي لمؤلف المهارة → الإعداد الافتراضي العام.

يأتي helm مزوّدًا بمهارات مدمجة للمهام التي ينفذها الوكلاء يوميًا: `fs` و`git` و`grep` و`edit` و`shell` و`http`. ويمكنك تعريف مهارات مخصّصة، فتحصل تلقائيًا على الأنواع، وإمكانية البحث، والأذونات.

## العرض التوضيحي

أنشأتُ روبوت دردشة يملك فيه الوكيل أداتين فقط: `search` و`execute`.

تُجري `search` بحثًا بالكلمات المفتاحية عبر جميع عمليات helm المسجَّلة — ويستدعيها الوكيل لاكتشاف ما هو متاح والتعرّف على تواقيع الدوال:

```ts
agent.search("file read");
// → [{ qualifiedName: "fs.read",
//      description: "Read a file and return its content as a string",
//      signature: "(path: string) => Promise<{ content: string }>", ... }]
```

تأخذ `execute` شيفرة JavaScript اعتباطية وتنفّذها باستخدام واجهة برمجة تطبيقات وكيل helm. يكتب النموذج اللغوي الكبير شيفرة مثل:

```js
const { staged, unstaged, branch } = await agent.git.status();
return { branch, staged: staged.length, unstaged: unstaged.length };
```

أداتان فقط ضمن السياق، بغضّ النظر عن عدد المهارات المسجّلة. يكتشف الوكيل ما يحتاج إليه عند الطلب ويكتب الشيفرة اللازمة لاستخدامه.

![يعرض تطبيق helm التجريبي الملفات في الدليل الحالي](/blog-images/helm-demo-app-list-files.png)

### عزل الشيفرة غير الموثوقة

تشغّل أداة `execute` أي شيفرة JavaScript يكتبها نموذج اللغة الكبير. ولجعل ذلك آمنًا، يعزلها العرض التوضيحي باستخدام [SES (Secure ECMAScript)](https://github.com/endojs/endo/tree/master/packages/ses) داخل عملية فرعية.

تقوم `lockdown()` في SES بتجميد جميع البُنى الجوهرية في JavaScript — مثل `Object` و`Array` و`Promise` و`Function`، وكل ما عدا ذلك. وتعمل الشيفرة داخل `Compartment`، وهو نطاق عام معزول لا يمكنه الوصول إلا إلى شيئين فقط: وكيل `agent` وواجهة `console` بديلة. أما `fetch` و`require` و`import` و`process` و`fs` — فلا وجود لأيٍّ منها داخل `Compartment`. والطريقة الوحيدة لفعل أي شيء مفيد هي عبر وكيل `agent`.

أما `agent` داخل بيئة العزل فليس وكيل helm الحقيقي — بل هو `Proxy` تكراري يعترض الوصول إلى الخصائص واستدعاءات الدوال. فعندما تستدعي الشيفرة `agent.git.status()`، يرسل هذا الـ `Proxy` رسالة IPC إلى العملية الأب. ثم تستدعي العملية الأب الدالة الحقيقية على وكيل helm الحقيقي، وتُجري فحص الأذونات الكامل، ثم ترسل النتيجة مرة أخرى. وإذا كانت العملية مضبوطة على `"ask"`، فتتوقف العملية الأب مؤقتًا بانتظار موافقة المستخدم قبل الرد. وإذا كانت مضبوطة على `"deny"`، فينتقل الخطأ مجددًا عبر IPC.

الشيفرة المعزولة لا تدري أن أيًا من هذا يحدث. هي فقط ترى أن `await` يُعيد قيمة. والطريقة الوحيدة للتفاعل مع العالم الخارجي هي عبر عمليات helm المقيّدة بالأذونات.

### واجهة الأذونات

تحتوي واجهة الدردشة على شريط جانبي يسرد كل مهارة وكل عملية مسجّلة، ولكل منها خيار تبديل بين allow/ask/deny. ويصبح تغيير الإذن ساريًا اعتبارًا من الرسالة التالية.

![لوحة الأدوات مع عناصر تحكم بالأذونات لكل عملية](/blog-images/helm-demo-app.png)

عندما يصل النموذج اللغوي الكبير إلى عملية مضبوطة على `"ask"`، يرسل الخادم طلب موافقة بشكل متدفق إلى الواجهة الأمامية. ويظهر في استدعاء الأداة شريط مضمّن يحوي زري Allow وDeny. وينتظر الخادم عند `Promise<boolean>` إلى أن ينقر المستخدم أحدهما.

![استدعاء أداة تنفيذ بانتظار موافقة المستخدم مع زري Allow وDeny](/blog-images/helm-demo-app-sandbox-feature.png)

إذا رفض المستخدم، فإن `PermissionDeniedError` تنتقل عبر السلسلة كاملةً، ويرى النموذج اللغوي الكبير ذلك في نتيجة الأداة. ويمكنه أن يشرح سبب حاجته إلى الإذن، أو يجرّب نهجًا مختلفًا، أو يتراجع.

## الإلهام

استُلهمت هذه المعمارية — التي تمنح الوكيل أداةً لتنفيذ الشيفرة بدلًا من عشرات الأدوات المنفصلة — من [وضع الكود](https://blog.cloudflare.com/code-mode-mcp/) في خادم MCP لدى Cloudflare، حيث خفّضوا استهلاك التوكنات بنسبة 99.9% عبر استبدال أكثر من 2,500 أداة لنقاط نهاية API بـ `search` + `execute`. وقد بلورت [فكرة Rhys Sullivan المشابهة](https://x.com/RhysSullivan/status/2019819177473933404) هذه الفكرة بالنسبة لي: فالجمع بين تنفيذ الشيفرة، وقابلية الاكتشاف، ونموذج أذونات دقيق يعني أن الوكيل يستطيع فعل أي شيء من دون أن ينفلت عن المسار.
