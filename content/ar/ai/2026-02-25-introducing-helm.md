# تقديم helm

helm هو إطار عمل TypeScript مضبوط الأنواع لوكلاء الذكاء الاصطناعي. بدلًا من استدعاء أوامر الصدفة وتحليل السلاسل النصية، تستدعي وكلاء الذكاء الاصطناعي دوالًا مضبوطة الأنواع بمدخلات ومخرجات منظَّمة:

```ts
import { createHelm, git, fs, grep } from "@bgub/helm";

const agent = createHelm({
  permissions: {
    "fs.read": "allow",
    "fs.write": "ask",
    "fs.remove": "deny",
    "git.status": "allow",
    "git.*": "ask",
  },
  onPermissionRequest: async (operation, args) => {
    return confirm(`Allow ${operation}?`);
  },
})
  .use(fs())
  .use(git())
  .use(grep());

const { staged, unstaged, branch } = await agent.git.status();
const { content } = await agent.fs.read("./package.json");
```

تُسجِّل &quot;المهارات&quot; — وهي مجموعات من العمليات المرتبطة — باستخدام نمط builder. يستنتج TypeScript النوع الكامل في كل خطوة. لكل عملية مستوى أذونات: `allow` (تعمل فورًا)، أو `ask` (تتوقف بانتظار الموافقة)، أو `deny` (يرمي `PermissionDeniedError`). تُحسَم الأذونات وفق ترتيب الأولوية: تطابق تام → wildcard → الإعداد الافتراضي لمؤلف المهارة → الإعداد الافتراضي العام.

يأتي helm مزوّدًا بمهارات مدمجة للمهام التي ينفذها الوكلاء كل يوم: `fs` و`git` و`grep` و`edit` و`shell` و`http`. يمكنك تعريف مهارات مخصصة، وستحصل تلقائيًا على الأنواع و`search` والأذونات.

## العرض التوضيحي

أنشأتُ روبوت دردشة لا يملك فيه الوكيل سوى أداتين فقط: `search` و`execute`.

تُجري `search` بحثًا بالكلمات المفتاحية عبر جميع عمليات helm المسجَّلة — ويستدعيها الوكيل لاكتشاف ما هو متاح ومعرفة تواقيع الدوال:

```ts
agent.search("file read");
// → [{ qualifiedName: "fs.read",
//      description: "Read a file and return its content as a string",
//      signature: "(path: string) => Promise<{ content: string }>", ... }]
```

تأخذ `execute` أي شيفرة JavaScript وتُشغّلها عبر واجهة برمجة التطبيقات الخاصة بوكيل helm. ويكتب النموذج اللغوي الكبير شيفرة مثل:

```js
const { staged, unstaged, branch } = await agent.git.status();
return { branch, staged: staged.length, unstaged: unstaged.length };
```

أداتان فقط ضمن السياق، بغضّ النظر عن عدد المهارات المسجَّلة. يكتشف الوكيل ما يحتاج إليه عند الحاجة ويكتب شيفرة لاستخدامه.

### عزل التعليمات البرمجية غير الموثوق بها

تشغّل أداة `execute` أي شيفرة JavaScript يكتبها النموذج اللغوي الكبير. ولجعل ذلك آمنًا، يعزلها العرض التوضيحي داخل بيئة sandbox باستخدام [SES (Secure ECMAScript)](https://github.com/endojs/endo/tree/master/packages/ses) في عملية فرعية.

تقوم `lockdown()` في SES بتجميد جميع الكيانات الجوهرية في JavaScript — `Object` و`Array` و`Promise` و`Function`، وغيرها كلها. وتعمل الشيفرة داخل `Compartment`، وهو نطاق عام معزول لا يتيح الوصول إلا إلى شيئين فقط: وكيل `agent` و`console` وهمي. أما `fetch` و`require` و`import` و`process` و`fs`، فلا وجود لأيٍّ منها داخل الـ `Compartment`. والطريقة الوحيدة لفعل أي شيء مفيد هي عبر وكيل `agent`.

أما `agent` داخل بيئة العزل فليس وكيل helm الحقيقي، بل `Proxy`递ادي يعترض الوصول إلى الخصائص واستدعاء الدوال. فعندما تستدعي الشيفرة `agent.git.status()`، يرسل الـ `Proxy` رسالة IPC إلى العملية الأب. ثم تستدعي العملية الأب الدالة الحقيقية على وكيل helm الحقيقي، وتجري فحص الأذونات الكامل، ثم ترسل النتيجة مرة أخرى. وإذا كانت العملية مضبوطة على `"ask"`، تُعلَّق العملية الأب حتى يوافق المستخدم قبل أن ترد. وإذا كانت مضبوطة على `"deny"`، فيُعاد تمرير الخطأ عبر IPC.

الشيفرة المعزولة لا تدرك أن شيئًا من هذا يحدث. كل ما تراه هو أن `await` يُرجع لها قيمة. والطريقة الوحيدة للتفاعل مع العالم الخارجي هي عبر عمليات helm المقيّدة بالأذونات.

### واجهة الأذونات

تتضمن واجهة الدردشة شريطًا جانبيًا يعرض كل `skill` وعملية مسجّلة، ولكلٍ منها مفتاح تبديل بين allow/ask/deny. ويسري تغيير أي إذن اعتبارًا من الرسالة التالية.

![لوحة الأدوات مع عناصر تحكم الأذونات لكل عملية](/blog-images/helm-demo-app.png)

عندما يصل النموذج اللغوي الكبير إلى عملية مضبوطة على `"ask"`، يبثّ الخادم طلب موافقة إلى الواجهة الأمامية. ويظهر استدعاء الأداة مع شريط تنبيه مضمّن يضم زري Allow وDeny. ويعلّق الخادم التنفيذ عند `Promise<boolean>` إلى أن ينقر المستخدم أحد الزرين.

![استدعاء أداة execute بانتظار موافقة المستخدم مع زري Allow وDeny](/blog-images/helm-demo-app-sandbox-feature.png)

إذا رفض المستخدم، فإن `PermissionDeniedError` ينتقل حتى يعود بالكامل، ويراه النموذج اللغوي الكبير في نتيجة الأداة. ويمكنه أن يشرح سبب حاجته إلى الإذن، أو يجرّب نهجًا مختلفًا، أو يتراجع.

## الإلهام

استُلهم هذا التصميم المعماري — القائم على تزويد الوكيل بأداة لتنفيذ الشيفرة بدلًا من عشرات الأدوات المنفصلة — من [code mode](https://blog.cloudflare.com/code-mode-mcp/) لدى Cloudflare لخادم MCP الخاص بها، حيث خفّضوا استهلاك التوكنات بنسبة 99.9% عبر استبدال أكثر من 2,500 أداة لنقاط نهاية واجهة برمجة التطبيقات بـ `search` و`execute`. وكانت [الفكرة المشابهة التي طرحها Rhys Sullivan](https://x.com/RhysSullivan/status/2019819177473933404) هي ما بلور الفكرة في ذهني: فالجمع بين تنفيذ الشيفرة، وإمكانية الاكتشاف، ونموذج أذونات دقيق يعني أن الوكيل يستطيع فعل أي شيء، من دون أن يخرج عن المسار.
