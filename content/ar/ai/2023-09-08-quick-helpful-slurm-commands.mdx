في المختبر الذي أعمل فيه، لدينا إمكانية الوصول إلى بيئة حوسبة عالية الأداء (HPC) تستخدم [Slurm Workload Manager](https://slurm.schedmd.com/documentation.html).

أستخدمه منذ مدة، وقد وجدت بعض الأوامر التي أرجع إليها باستمرار. فقلت أشاركها هنا لعلها تكون مفيدة لغيري أيضًا.

## التحقّق من حالة المهمة

```bash
# عرض جميع المهام
squeue
# التحقق من حالة مهامك أنت فقط
squeue -u <username>
# التحقق من حالة المهام بـ QOS محدد
squeue -q <QOS>
```

## إلغاء المهام

```bash
# إلغاء مهمة محددة
scancel <job_id>
# إلغاء جميع مهامك
scancel -u <username>
```

## طلب عقدة تفاعليًا

```bash
# طلب عقدة واحدة (سيفتح هذا جلسة تفاعلية في طرفيتك)
# عند الخروج من الطرفية، سيتم إعادة تخصيص العقدة
salloc --nodes=1 --gpus=8 --qos=<QOS> --mem=2000G --time=72:00:00 --ntasks=1 --cpus-per-task=128
```

## إرسال مهمة

ماذا لو كانت جميع عقد الحوسبة لديك مشغولة بالفعل، أو كنت لا تريد أن تنتهي مهمتك بمجرد إغلاق اتصال الطرفية؟ في هذه الحالة، يمكنك استخدام `sbatch` لإرسال مهمة إلى قائمة الانتظار. وستعمل تلقائيًا فور توفّر الموارد اللازمة.

سيتطلّب هذا إعدادًا إضافيًا بسيطًا. افترض أن المهمة التي نريد تشغيلها فعليًا موجودة في `myjob.sh`. ولكي نرسل هذا السكربت كمهمة، سننشئ أولًا سكربت Bash سيُشغَّله Slurm. لنسمّه `run.sh`:

```bash
#!/bin/bash
#SBATCH -J "JOBNAME"
#SBATCH --nodes=1
#SBATCH --gpus-per-node=8
#SBATCH --cpus-per-task=128
#SBATCH --mem=2000G
#SBATCH --time=72:00:00
#SBATCH --qos=<QOS>

srun --nodes=1 myjob.sh
```

لاحظ أننا نستخدم توجيه `#SBATCH` لتمرير المعاملات التي كنّا سنمرّرها سابقًا إلى `salloc`. كما نستخدم `srun` لتشغيل مهمتنا الفعلية؛ إذ سيتولى تشغيل السكربت عبر عدة عُقد إذا أردنا ذلك.

أخيرًا، لتشغيل السكربت، سننفّذ:

```bash
sbatch run.sh
```
