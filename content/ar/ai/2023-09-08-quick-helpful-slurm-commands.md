في المختبر الذي أعمل فيه، لدينا إمكانية الوصول إلى بيئة حوسبة عالية الأداء تستخدم [Slurm Workload Manager](https://slurm.schedmd.com/documentation.html).

أستخدمه منذ مدة، وقد اكتشفت بعض الأوامر التي أرجع إليها باستمرار. فقلت إنني سأشاركها هنا لعلّها تفيد غيري أيضًا.

## التحقق من حالة المهمة

```bash
# عرض جميع المهام
squeue
# التحقق من حالة مهامك فقط
squeue -u <username>
# التحقق من حالة المهام ذات QOS محدد
squeue -q <QOS>
```

## إلغاء المهام

```bash
# إلغاء مهمة محددة
scancel <job_id>
# إلغاء جميع مهامك
scancel -u <username>
```

## طلب عقدة تفاعليًا

```bash
# طلب عقدة واحدة (سيفتح هذا الأمر جلسة تفاعلية في طرفيتك)
# عند الخروج من الطرفية، سيتم إعادة تخصيص العقدة
salloc --nodes=1 --gpus=8 --qos=<QOS> --mem=2000G --time=72:00:00 --ntasks=1 --cpus-per-task=128
```

## إرسال مهمة

ماذا لو كانت جميع عُقد الحوسبة لديك محجوزة، أو كنت لا تريد أن تنتهي مهمتك فور إغلاق اتصالك بالطرفية؟ في هذه الحالة، يمكنك استخدام `sbatch` لإرسال مهمة إلى قائمة الانتظار. وستعمل تلقائيًا بمجرد توفّر الموارد اللازمة لها.

سيتطلب هذا بعض الإعداد الإضافي البسيط. افترض أن المهمة التي نريد تشغيلها فعليًا موجودة في `myjob.sh`. ولكي نرسل هذا السكربت كمهمة، سننشئ أولًا سكربت Bash سيشغّله Slurm. لنسمّه `run.sh`:

```bash
#!/bin/bash
#SBATCH -J "JOBNAME"
#SBATCH --nodes=1
#SBATCH --gpus-per-node=8
#SBATCH --cpus-per-task=128
#SBATCH --mem=2000G
#SBATCH --time=72:00:00
#SBATCH --qos=<QOS>

srun --nodes=1 myjob.sh
```

لاحظ أننا نستخدم توجيه المعالجة `#SBATCH` لتمرير المعلَمات التي كنّا سنمرّرها سابقًا إلى `salloc`. كما نستخدم `srun` لتشغيل المهمة الفعلية؛ إذ سيتولى تشغيل السكربت عبر عدة عُقد إذا رغبنا في ذلك.

أخيرًا، لتشغيل السكربت، سننفّذ:

```bash
sbatch run.sh
```
